# 🗺️ NexusMind: Cypher Query Playbook & Diagnostic Manual

This manual contains production-grade Cypher scripts for auditing, testing, and visualizing the dual-engine **Hierarchical GraphRAG** topology inside your local Neo4j Instance (`http://localhost:7474`).

---

## 📐 1. Graph Database Schema Overview

The database normalizes structural text geometries into uniform node classifications:
* **`DocumentNode`**: Universal label holding layout primitives. Classified into roles via internal properties:
  * `type: 'ParentChunk'` (Holds broad context frame strings (~2,000 characters))
  * `type: 'ChildChunk'` (Holds micro-fragment layout strings (~400 characters))
* **`CONCEPT`**: Abstract underlying domain theories, mathematical variables, and metric indices.
* **`TECHNOLOGY`**: Actionable software engineering frameworks, explicit algorithmic formulas, and tools.
* **`SYSTEM`**: Multi-component infrastructural software environments or system pipelines.

### Active Relationship Edge Network



* `(:DocumentNode {type: 'ChildChunk'}) -[:CHILD_OF]-> (:DocumentNode {type: 'ParentChunk'})`
* `(:CONCEPT|TECHNOLOGY|SYSTEM) -[:MENTIONED_IN]-> (:DocumentNode {type: 'ChildChunk'})`

## 📊 2. System Diagnostic & Schema Integrity Audits

Run these queries to verify that ingestion transactions are committing balanced data, mapping clean types, and identifying string encoding bugs.

### A. Core Node Count Matrix
Returns a flat analytical log summary of how many active nodes have been generated across all taxonomies:
```cypher
MATCH (n)
RETURN labels(n)[0] AS NodeLabel, count(n) AS NodeCount
ORDER BY NodeCount DESC;

```

### B. Segment Role Distribution Audit

Verifies that the `pdf_processor` chunking layout limits split structural nodes correctly between child snippets and their parent chapters:

```cypher
MATCH (d:DocumentNode)
RETURN d.type AS LayoutRole, count(d) AS Count, avg(size(d.text)) AS AvgCharacterLength
ORDER BY Count DESC;

```

### C. Floating Entity Error Scan

Finds broken graph threads where an entity node was generated by the multi-agent pass but failed to anchor down to a matching vector document fragment block:

```cypher

MATCH (e)
WHERE (e:CONCEPT OR e:TECHNOLOGY OR e:SYSTEM) 
  AND NOT (e)-[:MENTIONED_IN]->(:DocumentNode)
RETURN e.id AS TrueFloatingEntityID, labels(e)[0] AS LabelType
LIMIT 50;

```

## 🔍 3. Live GraphRAG Retrieval Traversal Queries

These queries mirror the exact multi-hop multi-layer context lookups performed by your active backend server execution thread during runtime lookups.

### A. The Master 3-Tier Multi-Hop Path (Visualizer Mainstay)

Pulls back the complete contextual path of your database. Use this query to inspect your structural clusters:

```cypher
MATCH (entity)-[r1:MENTIONED_IN]->(child:DocumentNode)-[r2:CHILD_OF]->(parent:DocumentNode)
WHERE child.type = 'ChildChunk' AND parent.type = 'ParentChunk'
RETURN entity, r1, child, r2, parent
LIMIT 50;

```

### B. Dense Keyword Convergence Search

Finds hot spots in your graph—specific document sections where multiple advanced concepts overlap. This shows exactly which text frames hold the highest information density:

```cypher
MATCH (entity)-[:MENTIONED_IN]->(child:DocumentNode)
WITH child, count(entity) AS ConceptDensity, collect(entity.id) AS SharedConcepts
WHERE ConceptDensity > 2
MATCH (child)-[:CHILD_OF]->(parent:DocumentNode)
RETURN child.id AS ChunkID, ConceptDensity, SharedConcepts, parent.id AS ParentID
ORDER BY ConceptDensity DESC
LIMIT 10;

```

### C. Entity Co-Occurrence Discovery

Analyzes cross-chapter conceptual linkages. It returns which independent technologies or concepts are frequently discussed alongside each other across separate paragraphs:

```cypher
MATCH (a)-[:MENTIONED_IN]->(c:DocumentNode)<-[:MENTIONED_IN]->(b)
WHERE id(a) < id(b) AND (a:CONCEPT OR a:TECHNOLOGY OR a:SYSTEM) AND (b:CONCEPT OR b:TECHNOLOGY OR b:SYSTEM)
RETURN a.id AS EntityA, labels(a)[0] AS TypeA, b.id AS EntityB, labels(b)[0] AS TypeB, count(c) AS CoOccurrenceStrength
ORDER BY CoOccurrenceStrength DESC
LIMIT 15;

```

## 🧹 4. Maintenance, Deduplication & Administrative Operations

Use these queries to manage structural changes, run quick lookups, or completely reset your development environment.

### A. Targeted Entity Lookup & Path Deletion

Safely isolates a misspelled entity or hallucinated term and removes it from the network without damaging your core document nodes:

```cypher
MATCH (e {{id: "MALFORMED_ENTITY_NAME"}})
DETACH DELETE e;

```

### B. Smooth Dynamic Node Re-Labeling

If your model misclassifies an entity (e.g., tagging a concept as a technology), this query safely re-assigns the node type:

```cypher
MATCH (e:TECHNOLOGY {id: "DIMENSIONALITY REDUCTION"})
REMOVE e:TECHNOLOGY
SET e:CONCEPT
RETURN e;

```

### C. The Absolute Database Wipe (Nuclear Option)

Completely flushes out all graph structures, relationships, properties, and constraint historical schemas. Run this before executing a fresh pipeline pass:

```cypher
MATCH (n)
DETACH DELETE n;

### 💡 Pro-Tip for Neo4j Browser Layouts

When inspecting **Query 3A** inside your visualizer window:
1. Go to the right-side properties toolbar and click on the **`DocumentNode`** label token bubble.
2. At the bottom of your graph pane, switch the node display property from `id` to **`type`**. 
3. Next, select the **`CONCEPT`** and **`TECHNOLOGY`** label tags and set their visual display properties to **`id`**. 

This will automatically clean up your graph canvas, replacing long, cluttered text blocks with clear, scannable technical terms!
